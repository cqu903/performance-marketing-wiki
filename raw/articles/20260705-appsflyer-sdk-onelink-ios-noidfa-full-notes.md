# 1. AppsFlyer基础工作原理：SDK + 末次点击归因逻辑
AppsFlyer 的运转依靠两大核心组件：
1. **App内集成SDK**
   负责监听、采集：App安装激活、首次打开、自定义应用内转化事件，用户打开App时自动向AF上报install激活事件与设备信息。
2. **归因链接（Tracking Link / OneLink）**
   每个渠道、每条广告单独分配带渠道参数的专属追踪链接；用户触发点击时，渠道、活动、素材、设备弱指纹等信息同步记录到AppsFlyer后台。

## 归因判定规则：末次点击 Last-Click
1. 发生App安装激活时，AF检索归因窗口期内所有已记录的用户点击触点；
2. 筛选出**时间距离激活最近**的一条点击，判定为本次安装的归属渠道；
3. 若窗口期内仅广告曝光、无任何点击，则启用展示归因VTA补充规则。

> 原文关键定义补充：归因链接的数据上报行为，默认指代**广告素材本身的点击**，并非落地页H5内部按钮的二次点击。

# 2. 归因链接上报时机：广告点击 vs H5二次按钮点击区分
## 结论
标准投放链路下：**归因链接向AF上报点击事件的时机 = 用户点击广告素材的瞬间**；
广告跳转H5、再点击H5下载按钮属于二次CTA，分两种配置场景。

## 场景1：广告绑定原生AF链接，H5按钮无独立OneLink（主流）
1. 用户点击广告素材，瞬间请求AF服务器，上传设备、渠道参数，AF生成并存储一条点击触点；
2. AF 302重定向至H5落地页，归因参数拼接在H5 URL中；
3. 用户浏览H5后点击页面下载按钮，按钮仅做普通跳转商店，**不会新增AF点击上报**；
4. 整条用户路径仅存在1条广告点击记录，最终安装全部归因给本次广告点击。

## 场景2：H5下载按钮单独挂载独立AF OneLink（再营销/分层追踪场景）
1. 第一次点击广告：上报第一条点击触点；
2. 用户在H5页面点击带独立OneLink的下载按钮，**再次触发一次全新点击上报，生成第二条触点记录**；
3. 激活发生后，系统遵循末次点击规则，时间更晚的H5按钮点击会抢占归因归属。

## 操作行为对照表
| 用户操作 | 是否上报AF点击 | 是否生成独立触点 | 末次归因优先级 |
|--------|--------------|--------------|------------|
| 直接点击广告直跳商店 | 是，点击瞬间上报 | 1条唯一触点 | 唯一触点，安装归该广告 |
| 点广告跳H5，按钮无独立AF链接 | 仅广告点击上报 | 仅广告1条触点 | 归属原始广告点击 |
| 点广告跳H5，按钮绑定独立OneLink | 广告、H5按钮两次上报 | 2条时间不同触点 | H5按钮点击时间更晚，抢占归因 |

# 3. 广告原生AF链接直跳商店 / 跳转H5落地页归因差异
## 表层相同点
广告仅挂载一套原生AF链接、H5无独立追踪链接时：
1. AF数据库仅留存1条广告点击触点；
2. 末次点击判定结果、归因窗口计时起点、渠道转化回传逻辑完全一致；
3. 窗口统一从广告点击瞬间开始计时，和H5停留时长无关。

## 核心归因维度差异（抛开用户体验）
| 对比维度 | 标准直跳商店链路 | 广告OneLink跳转自定义H5落地页 |
|--------|----------------|--------------------------|
| 无效点击损耗 | 极低，无中间页面流失 | 高，大量只浏览H5不下载的无效点击，拉低渠道CVR |
| 网页行为埋点能力 | 完全不支持，无法统计网页表单、停留、按钮点击 | 支持，搭配AF Smart Script可搭建广告→网页→App完整漏斗 |
| Web浏览归因(VTA) | 无法开启 | 可开启曝光概率归因，仅浏览广告未点击也可做归因兜底 |
| iOS SKAdNetwork稳定性 | 最优，无网页跳转干扰SKAN注册流程 | H5加载延迟会小幅提升SKAN注册失败概率 |
| 网页再营销分层 | 不支持无网页参数缓存 | 可缓存af_clickid，基于原始广告参数做二次人群分发 |

## 风险场景补充
1. 用户点广告进H5直接关闭，3天后主动商店搜索下载：两种链路均可正常归因广告；
2. 需要统计网页留资、预约等前置转化行为：仅H5落地页链路支持；
3. H5白屏、加载失败退出：仅产生无效点击，不破坏原始广告点击记录，后续下载仍可正常归因。

# 4. 落地页场景：商店无任何参数时，AF如何捕获install安装数据
## 核心逻辑
归因的全部设备与渠道证据，**在广告点击阶段就永久存入AF后台**；商店页面不会、也不需要传递任何归因参数，安装匹配依靠两套独立数据流完成绑定。

## 完整分步流程（广告→H5→商店→安装激活）
### 步骤1：点击广告，AF永久存储点击触点
1. 用户点击广告OneLink，浏览器请求AF服务器；
2. AF立即写入click记录，存储内容：渠道pid、campaign、素材、全局唯一`af_clickid`、IP、UA、精确点击时间；
3. AF重定向H5页面，全部归因参数拼接在H5 URL上。

### 步骤2：H5点击下载按钮跳转商店（商店无AF参数）
分两种H5配置：
1. **无AF Smart Script**：按钮纯普通商店链接，跳转后商店URL干净，无任何追踪参数；
2. **部署AF Smart Script**：脚本读取URL内`af_clickid`存入浏览器localStorage缓存；可选动态生成新OneLink跳转商店，新增第二条点击触点；
两种场景下，App Store/Google Play本身均不会携带归因参数。

### 步骤3：App首开，SDK上报install激活事件（AF拿到安装数据）
App内置AF SDK，用户首次打开App自动执行采集上报，和商店无任何交互：
1. SDK采集本机设备标识、IP、机型、系统版本；
2. 封装install事件发送至AF服务器；
3. AF后台启动点击记录与安装设备的匹配逻辑。

### 步骤4：AF多层匹配绑定点击与激活
1. **Android一级精准匹配**：Google Install Referrer，读取商店referrer内`af_clickid`直接精准匹配；
2. **iOS有IDFA场景精准匹配**：点击时媒体传递的IDFA与SDK上报IDFA完全一致，一键匹配；
3. **兜底概率指纹匹配**：IDFA失效/无referrer时，依靠IP、机型、系统、时间窗口多维度组合匹配；
4. 匹配成功则标记渠道归属，无匹配标记为自然量organic。

## 关键总结
商店仅作为App下载载体，不参与归因线索存储；所有匹配依据来自「广告点击时存入后台的设备指纹」与「App打开后SDK上报的设备信息」双向校验。

# 5. 用户点击广告的时候，浏览器/H5能否拿到IDFA
## 核心结论
1. Safari、H5网页、浏览器JS**完全没有系统权限读取本机IDFA**；IDFA仅iOS原生App可通过系统框架获取；
2. ATT追踪弹窗仅在App首次打开时弹出，点击广告阶段App尚未安装，不存在授权读取IDFA的条件；
3. AF服务器无法主动抓取IDFA，仅能被动接收广告媒体App拼接在OneLink内的IDFA参数。

## 两种点击环境区分
### 环境A：在抖音/Meta/快手等广告媒体App内点击广告
1. 媒体App已获取用户ATT授权，可读取本机IDFA；
2. 跳转AF OneLink时，媒体自动追加`&idfa=xxxx`参数传入链接；
3. AF服务器存储点击记录时可留存IDFA；
4. 若用户选择拒绝追踪，媒体传入的IDFA为全零无效字符串。

### 环境B：短信外链、Safari直接打开OneLink（纯浏览器环境）
无媒体App读取IDFA，URL不会携带任何idfa字段，AF点击记录仅留存IP、UA、时间等弱指纹。

## 阶段区分：点击链路 vs App激活链路
1. 广告点击阶段：网页无读取IDFA能力，仅媒体App可被动传入；
2. App首开激活阶段：AF SDK是全链路唯一稳定、可靠获取IDFA的渠道，绝大多数iOS精准归因依靠SDK上报的IDFA完成匹配。

## H5落地页补充
即便部署AF Smart Script网页脚本，也仅能读取URL内渠道参数、缓存`af_clickid`，依旧无法获取本机IDFA。

# 6. iOS用户关闭IDFA（ATT选择不允许追踪）完整归因工作流程
## 前置状态
用户拒绝追踪后，AF SDK读取IDFA为全零字符串，传统ID精准匹配失效；AF启用四层优先级并行归因方案，全程遵守末次点击规则。

## 阶段1：广告点击环节数据情况
1. H5/浏览器无法读取IDFA；媒体App传入的idfa为全零，无效；
2. AF点击记录仅保留有效信号：渠道参数、唯一`af_clickid`、IP、UA、时区、点击时间戳；
3. H5 Smart Script可缓存`af_clickid`至浏览器本地存储。

## 阶段2：App首开install上报数据情况
1. SDK无法获取有效IDFA；
2. 可上报信号：IDFV、当前IP、机型、系统版本、激活时间、网页缓存带回的`af_clickid`；
3. 无跨设备唯一广告标识符用于精准匹配。

## 四层归因匹配机制（优先级从高到低）
### 第一层：SKAdNetwork（SKAN，苹果官方隐私归因，最高优先级）
1. 媒体App点击广告时调用苹果SKAN接口，向系统报备广告渠道、活动签名；
2. 用户完成安装后，苹果系统延迟24~64小时自动匿名推送转化数据给AF；
3. 回传内容仅包含广告网络ID、活动ID、转化值，无任何用户设备标识；
4. 优点：100%合规、无误匹配；缺点：数据延迟高、仅支持活动层级，无法细分素材、晚转化丢失。

### 第二层：af_clickid精准匹配（落地页必备高精度方案）
生效前提：媒体App内点击广告 + H5部署AF Smart Script缓存`af_clickid`
1. 广告点击生成全局唯一`af_clickid`，网页脚本存入localStorage；
2. App打开后SDK读取缓存的`af_clickid`随install事件上报；
3. AF后台直接通过`af_clickid`精准命中对应点击记录，不受IDFA影响，匹配准确率100%；
4. 纯浏览器打开链接、无H5缓存时该链路失效。

### 第三层：IDFV弱关联补充（几乎不适用于外部买量）
IDFV为单开发者名下App共享设备标识符，跨媒体、跨开发者渠道无法匹配；仅自有多App内部流量可辅助串联用户，投放买量场景基本无价值。

### 第四层：概率指纹建模兜底（上层匹配全部失效时启动）
匹配使用多维弱信号交集：
1. 核心信号：公网IP；辅助信号：机型、iOS版本、时区、语言；
2. 约束条件：激活时间必须在7天归因窗口内；多条匹配记录时选取**末次点击**；
3. 判定逻辑：多维信号重合计算匹配置信分，达到阈值则匹配成功；置信度过低标记为自然量；
4. 短板：公共WiFi同IP易出现误匹配、动态IP切换导致归因丢失。

## 带H5落地页完整闭环案例（用户拒绝IDFA）
链路：抖音广告 → AF OneLink跳转自有H5落地页 → H5按钮跳转App Store → 安装后ATT拒绝追踪
1. 点击阶段：抖音媒体传入全零IDFA，AF存储点击记录，H5脚本缓存`af_clickid`；
2. 商店跳转：无任何归因参数；
3. App首开：SDK上报缓存的`af_clickid`，无有效IDFA；
4. AF匹配流程：优先等待SKAN回传，同步校验`af_clickid`直接精准命中广告点击，完成归因。

## 关键问答
1. 关闭IDFA后末次点击规则是否生效？
完全生效，SKAN、af_clickid匹配、概率指纹多触点场景均优先选择时间最近的点击作为归因赢家。
2. 直跳商店对比H5落地页，哪种无IDFA丢失更少？
H5落地页搭配Smart Script可缓存`af_clickid`，启用第二层精准匹配，归因丢失率显著低于直跳商店链路。
3. 安卓是否存在同类问题？
安卓无ATT框架，无法关闭GAID；依靠Google Install Referrer的clickid实现确定性匹配，指纹兜底丢失率远低于iOS。
